# 四项目 task-list 归纳标准

本文基于四组样本整理：

- PDF Markdown Skill：`docs/discussion/1-skills-pdf-markdown/task-list(pdf-markdown).md`
- PRD Workflow：`docs/discussion/2-prd-workflow/task-list(prd-workflow).md`
- Translation Workflow：`docs/discussion/3-translation-workflow/task-list(translation).md`
- Patent Workflow：`docs/discussion/4-patent-workflow/task-list(patent).md`

## 共同点

1. `task-list.md` 是项目级任务台账，不是随手日志。
2. 文件头承担轻量 schema 职责：说明记录范围、动作枚举、同义归并规则。
3. 动作字段统一收敛为 8 个值：`修复`、`开发`、`优化`、`调整`、`规划`、`检查`、`文档`、`运维`。
4. 主体结构是 `## 二级分类 + Markdown 表格`。
5. 基础字段都是 `ID / 动作 / 描述 / 发现时间 / 完成时间 / 状态 / 备注`。
6. Bug 类通常使用 `问题描述`，非 Bug 类通常使用 `事项`。
7. 时间字段统一拆成 `发现时间` 和 `完成时间`，格式为 `YYYY-MM-DD HH:MM`，使用机器本地时区的 24 小时制时间。
8. `备注` 是信息密度最高的字段，承载根因、实现摘要、影响文件、测试结果、审查来源和后续说明。
9. ID 采用 `PREFIX-三位编号`，要求递增、不复用、避免重复。

## 差异点

| 维度 | 差异 | 推荐处理 |
| --- | --- | --- |
| 分类命名 | `功能开发` vs `开发事项`，`文档维护` vs `文档事项` | 默认用 `功能开发` / `文档维护`；如已有项目用另一套命名，保留原习惯 |
| 分类集合 | 有的项目包含 `规划事项`、`优化事项`、`调研事项`、`配置运维` | 默认只启用常用 7 类；额外分类按需添加 |
| 开发字段 | 专利项目使用 `优先级`、`预计时间` 两个扩展列 | 仅在项目需要排期或 WBS 管理时启用 |
| 状态模型 | 有的只有终态，有的包含 `待修复`、`待开发` | 默认支持活跃状态，但不要强迫旧完成台账改造 |
| 统计摘要 | 专利项目有摘要，其他项目没有 | 作为可选扩展，不设为默认必需 |
| 历史质量 | PRD/专利样本存在重复 ID、重复章节、空表头、未转义竖线 | CLI 和人工检查都应覆盖这些风险 |

## 推荐统一模型

### 默认分区

| 分区 | 前缀 | 主要用途 |
| --- | --- | --- |
| 代码 Bug | `BUG` | 缺陷、回归、安全风险、错误行为 |
| 调整事项 | `ADJ` | 需求口径、项目定位、配置/规范调整 |
| 检查事项 | `CHK` | 审计、复核、验证、代码审查、评估 |
| 测试数据 | `TST` | 测试样本、夹具数据、测试输出、验证数据集 |
| 文档维护 | `DOC` | 文档新增、修订、归档、结构说明 |
| 功能开发 | `DEV` | 新功能、模块建设、工程能力 |
| 配置运维 | `OPS` | 环境、部署、依赖、Git、配置维护 |

### 可选分区

| 分区 | 前缀 | 启用条件 |
| --- | --- | --- |
| 规划事项 | `PLN` | 项目有路线图、阶段计划、WBS、方案沉淀 |
| 优化事项 | `OPT` | 优化/重构事项很多，不适合混在开发或 Bug 中 |
| 调研事项 | `RES` | 项目经常记录外部资料、竞品、开源项目或专利调研 |
| 统计摘要 | 无 | 项目需要显式追踪完成率或待处理数量 |

## 字段定义

### 通用 7 列

| 字段 | 定义 | 规则 |
| --- | --- | --- |
| `ID` | 任务唯一编号 | `<前缀>-<三位数字>`，如 `BUG-001` |
| `动作` | 事项性质 | 只能使用 8 个动作枚举 |
| `问题描述 / 事项` | 一句话描述 | Bug 用 `问题描述`，其他默认用 `事项` |
| `发现时间` | 事项被发现、提出或登记的时间 | 格式为 `YYYY-MM-DD HH:MM`，24 小时制，默认使用机器本地时区 |
| `完成时间` | 事项完成、修复或关闭的时间 | 格式为 `YYYY-MM-DD HH:MM`；未完成事项填 `-` |
| `状态` | 当前状态 | Bug 完成态用 `已修复`，非 Bug 完成态用 `已完成` |
| `备注` | 细节说明 | 写清根因、改动、文件、测试、来源 |

### 开发 9 列扩展

```markdown
| ID | 动作 | 事项 | 优先级 | 预计时间 | 发现时间 | 完成时间 | 状态 | 备注 |
```

适用于 WBS、专利工作流、长期工程计划等需要排期的项目。

## 状态建议

| 状态 | 适用场景 |
| --- | --- |
| `待修复` | Bug 已记录但尚未修复 |
| `已修复` | Bug 已修复并验证 |
| `待开发` | 非 Bug 任务尚未开始 |
| `进行中` | 已开始但未完成 |
| `已完成` | 非 Bug 任务完成 |
| `已解决` | 排查类事项已定位并闭环；新条目优先用 `已完成`，仅在明确描述故障排查闭环时使用 |
| `已关闭` | 作废、不处理、被其他条目替代 |

如果项目只想维护“完成台账”，可以只使用 `已修复` 和 `已完成`。`已解决`、`已修复`、`已完成`、`已关闭` 都按完成态统计。

## 记录方案

新增记录时按以下顺序处理：

1. 判断事项类型，选择分区。
2. 将动作归并到 8 个标准枚举之一。
3. 搜索当前前缀最大 ID，分配下一个三位编号。
4. 写一句可追踪的描述。
5. 填写发现时间，格式为 `YYYY-MM-DD HH:MM`，使用机器本地时区。
6. 填写完成时间；已完成事项使用 `YYYY-MM-DD HH:MM`，未完成事项用 `-`。
7. 填写真实状态。
8. 在 `备注` 写关键证据：文件、命令、测试、审查来源和后续风险。
9. 如有统计摘要，同步更新摘要。

## 已有列表标准化

`standardize` 用于已经运行一段时间的 `task-list.md`。它不是简单套模板，而是先诊断，再决定是否修复。

以下命令在 Windows 上把 `python3` 替换为 `python`。

默认只生成报告，不修改文件：

```bash
python3 skills/task-list-initialization/scripts/task_list_cli.py standardize --file task-list.md --report docs/task-list-standardize-report.md
```

报告应覆盖：

| 维度 | 检查内容 |
| --- | --- |
| 结构合规 | 重复 ID、重复章节、坏表格行、动作/状态枚举、表头是否符合当前 7 列时间模型 |
| 记录完整性 | 备注是否有证据、完成记录是否有完成时间、未完成记录是否误填完成时间 |
| 分类清晰度 | ID 前缀与所在章节是否一致，调研/优化/文档/运维事项是否混放 |
| Profile 建议 | 是否继续使用 `minimal`，或升级为 `planning`、`extended`、`development` |
| 可自动修复 | 缺失空分区、旧单日期 schema 迁移等低风险事项 |
| 需人工确认 | 重命名章节、移动记录、重写重复 ID、拆分语义混杂事项 |

修复必须显式开启：

| 参数 | 行为 |
| --- | --- |
| `--apply-safe-fixes` | 执行低风险修复，例如补齐缺失空分区 |
| `--migrate-schema` | 将旧 `发现日期/完成日期/规划日期` 单日期列迁移为 `发现时间/完成时间` |
| `--fix-only` | 输出修饰：仅打印修复摘要、不展开完整诊断报告；本身不触发修复，需配合 `--apply-safe-fixes` 或 `--migrate-schema` |

标准化时应保留历史事实。不要为了贴合模板自动改写事项含义、移动记录、合并重复 ID 或重算摘要，除非用户明确批准。

## 样式约定

- 使用 Markdown 表格，不使用 bullet list 代替正式记录。
- 每个分区只保留一张表。
- 表头后必须有分隔行。
- 表格单元格里的 `|` 必须写成 `\|`。
- 路径、命令、代码标识用反引号。
- 备注可以长，但应优先写结构化事实，不写空泛总结。

## 常见问题和处理

| 问题 | 处理 |
| --- | --- |
| 重复 ID | 不继续复用；如需清理，新增 `ADJ-` 记录说明改号映射 |
| 重复章节 | 合并到同名第一节，并保留条目原始顺序或按 ID 排序 |
| 空表头 | 删除多余表头，只保留一个有效表 |
| ID 前缀与章节不一致 | 新增记录必须一致；历史数据是否移动需用户确认 |
| 状态枚举混乱 | 先定义状态集合，再批量规范 |
| 备注破坏表格 | 转义 `|`，必要时把长说明拆成简洁事实 |
